# RetailCo Data Engineering Exercise

A hands-on Databricks exercise demonstrating core data engineering concepts:
raw data ingestion, data profiling, data quality remediation, and building
a star schema data mart for a reporting pipeline.

---

## Scenario

You are a data engineer at **RetailCo**, a European online retailer operating
in the UK, France, Germany, Spain, and Italy. The analytics team needs a
**sales data mart** to power BI dashboards tracking revenue, margins, and
customer behaviour.

Your job is to take three messy operational tables and transform them into a
clean, query-optimised star schema.

---

## Data Sources (Raw Layer)

| Table | Rows | Description |
|---|---|---|
| `raw.customers` | 500 | Customer master — name, email, country, city, segment |
| `raw.products` | 105 | Product catalogue — category, brand, pricing |
| `raw.orders` | 5 000 | Order transactions — date, quantity, price, discount, status |

> Raw tables are Delta Lake tables generated by `00_setup_data`.
> They intentionally contain data quality issues for you to discover and fix.

## Target Data Model (Star Schema)

```
                       ┌─────────────┐
                       │  dim_date   │
                       │─────────────│
                       │ date_key PK │
                       │ full_date   │
                       │ day_of_week │
                       │ month       │
                       │ quarter     │
                       │ year        │
                       │ is_weekend  │
                       └──────┬──────┘
                              │
┌──────────────┐     ┌────────┴─────────┐     ┌──────────────┐
│ dim_customer │     │   fact_sales     │     │ dim_product  │
│──────────────│     │──────────────────│     │──────────────│
│customer_key  │◄────│ customer_key  FK │     │ product_key  │
│customer_id   │     │ product_key   FK─┼────►│ product_id   │
│full_name     │     │ date_key      FK │     │ product_name │
│email         │     │ order_id        │     │ category     │
│country       │     │ quantity        │     │ subcategory  │
│city          │     │ unit_price      │     │ brand        │
│segment       │     │ discount_pct    │     │ cost_price   │
│signup_date   │     │ gross_amount    │     │ list_price   │
└──────────────┘     │ discount_amount │     └──────────────┘
                      │ net_amount      │
                      │ cost_amount     │
                      │ margin_amount   │
                      └─────────────────┘
```

### Financial Metrics in `fact_sales`

| Column | Formula |
|---|---|
| `gross_amount` | `quantity × unit_price` |
| `discount_amount` | `gross_amount × discount_pct / 100` |
| `net_amount` | `gross_amount − discount_amount` |
| `cost_amount` | `quantity × cost_price` (from dim_product) |
| `margin_amount` | `net_amount − cost_amount` |

## Data Quality Issues to Discover

Part of the exercise is finding and fixing these issues before building the mart:

| # | Issue | Table | Column | Action |
|---|---|---|---|---|
| Q1 | ~17 % null values | `raw.customers` | `city` | Replace with `"Unknown"` |
| Q2 | 5 duplicate rows | `raw.products` | `product_id` | Deduplicate (keep first) |
| Q3 | 5 rows with negative quantity | `raw.orders` | `quantity` | Exclude from fact |
| Q4 | ~2 % null discount | `raw.orders` | `discount_pct` | Impute with `0` |
| Q5 | ~15 orders, unknown customer | `raw.orders` | `customer_id` | Assign sentinel key `-1` |
| Q6 | Cancelled & Returned orders | `raw.orders` | `status` | Exclude from fact |

## Learning Objectives

After completing this exercise you will be able to:

1. **Profile raw data** — identify nulls, duplicates, outliers, and referential integrity breaks
2. **Apply data quality rules** — make and document remediation decisions
3. **Build a date dimension** — generate a calendar table programmatically with PySpark
4. **Apply surrogate keys** — use `row_number()` windows for deterministic integer keys
5. **Design a star schema** — separate measures (fact) from context (dimensions)
6. **Compute financial metrics** — chain column expressions for revenue, discount, margin
7. **Validate a data mart** — referential integrity checks and financial consistency checks
8. **Write reporting queries** — join across a star schema for BI-ready aggregations

## Reporting Use Cases Enabled by the Mart

Once built, the mart directly answers:

- Monthly and quarterly revenue trends
- Margin analysis by product category and subcategory
- Customer segment contribution (Premium / Standard / Basic)
- Country-level revenue breakdown
- Weekend vs weekday purchasing patterns
- Top customers by net revenue
- Brand performance within categories
